# 핑 멀티테넌시 데이터 구조 설계도

> 목적: 현재 "단일 가족(`FAMILY_ID='kimfamily'` 고정)" 구조를 **여러 가족이 각자 데이터를 안전하게 분리해서 쓰는** 구조로 전환하기 위한 설계.

작성: 김아빠 · 버전 0.1 · 2026-06-15

---

## 1. 현재(AS-IS) vs 목표(TO-BE)

| 항목 | 현재 | 목표 |
|---|---|---|
| 가족 식별 | 코드에 `FAMILY_ID='kimfamily'` 고정 | 가입 시 가족마다 고유 `familyId` 발급 |
| 구성원 | 이름 4명 하드코딩(아빠/엄마/윤아/건우) | Firebase Auth 계정(uid) = 구성원 |
| 데이터 위치 | `pings/{date_member}` 최상위 + `familyId` 필드 | `families/{familyId}/...` 하위로 격리 |
| 접근 제어 | 사실상 공개(누구나 같은 데이터) | 가족 구성원만 자기 가족 데이터 접근 |
| 가입/합류 | 없음 | 가족 생성 + 초대코드 합류 |
| 펫/보상/점수 | 코드 상수 | 가족별 문서로 커스터마이즈 |

**핵심 원칙**: 모든 가족 데이터는 `families/{familyId}` 아래에 둔다(테넌트 격리). 접근 권한은 "그 가족의 멤버인가?"로 판단한다.

## 2. 컬렉션 트리 (Firestore)

```
users/{uid}
  └─ defaultFamilyId, familyIds[], displayName, createdAt

inviteCodes/{code}                 # 초대코드 → 가족 매핑 (합류 조회용)
  └─ familyId, expiresAt, createdBy

families/{familyId}
  ├─ name, ownerUid, createdAt
  ├─ season: { length: '3개월', startedAt }
  ├─ scoreRules: { ping:10, allMembers:5, withPet:30, gathering:50,
  │               withPetWeekly:true, gatheringWeekly:true }
  │
  ├─ members/{uid}
  │     └─ displayName, role, avatarUrl, joinedAt, isOwner
  │
  ├─ pets/{petId}
  │     └─ name, kind(강아지/고양이), breed, birth, weightKg,
  │        sex, neutered, health[ '육류 알레르기','눈물' ], memo,
  │        mascotUrls: { happy, play, love, ... }
  │
  ├─ pings/{YYYY-MM-DD_uid}
  │     └─ memberUid, date, photoURL, comment, withPet(bool),
  │        familyGathering(bool), reactions:{uid:emoji}, question, createdAt
  │
  ├─ rewards/{rewardId}
  │     └─ icon, name, pt, detail, achievedAt(null|ts), order
  │
  ├─ pokes/{pokeId}
  │     └─ fromUid, targetUid, createdAt
  │
  └─ notifications/{notifId}
        └─ type, fromUid, body, createdAt, readBy[]

fcmTokens/{uid}                    # 서버(Functions)만 읽음
  └─ token, familyId, updatedAt
```

> 설계 메모: `pings` 문서 ID를 `날짜_uid`로 두면 "하루 1회"가 자연스럽게 보장된다(같은 키 덮어쓰기). 점수 집계는 클라이언트 또는 Functions에서 가족 단위로 수행.

## 3. 문서 스키마 상세

### families/{familyId}
| 필드 | 타입 | 설명 |
|---|---|---|
| name | string | 가족 방 이름 |
| ownerUid | string | 방장(생성자) uid |
| createdAt | timestamp | 생성 시각 |
| season | map | `{ length, startedAt }` 시즌 설정 |
| scoreRules | map | 점수/보너스 값·주간제한 플래그 |

### members/{uid}
| 필드 | 타입 | 설명 |
|---|---|---|
| displayName | string | 표시 이름(아빠 등) |
| role | string | 역할 |
| avatarUrl | string | 아바타 이미지 |
| joinedAt | timestamp | 합류 시각 |
| isOwner | bool | 방장 여부 |

### pets/{petId}
| 필드 | 타입 | 설명 |
|---|---|---|
| name | string | 이름(뚜비) |
| kind / breed | string | 종류 / 품종 |
| birth | string | 생일(YYYY-MM-DD) |
| weightKg | number | 체중 |
| sex / neutered | string / bool | 성별 / 중성화 |
| health | array | 알레르기·특이사항 (맞춤 추천 키) |
| memo | string | 메모 |
| mascotUrls | map | 표정별 마스코트 이미지 URL |

### pings/{date_uid}
| 필드 | 타입 | 설명 |
|---|---|---|
| memberUid | string | 올린 사람 |
| date | string | YYYY-MM-DD |
| photoURL | string | 사진 |
| comment | string | 한 줄 설명(≤20자) |
| withPet | bool | 반려동물과 함께(주1회 보너스) |
| familyGathering | bool | 온 가족 모임(주1회 전원 보너스) |
| reactions | map | `{uid: emoji}` |
| createdAt | timestamp | 작성 시각 |

## 4. 보안 규칙 (firestore.rules)

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() { return request.auth != null; }
    function uid() { return request.auth.uid; }
    function isMember(fid) {
      return signedIn() &&
        exists(/databases/$(database)/documents/families/$(fid)/members/$(uid()));
    }
    function isOwner(fid) {
      return signedIn() &&
        get(/databases/$(database)/documents/families/$(fid)).data.ownerUid == uid();
    }

    // 사용자 본인 문서
    match /users/{userId} {
      allow read, write: if signedIn() && userId == uid();
    }

    // 초대코드: 코드를 아는 사람만 단건 조회(합류용). 목록·생성은 막음(서버에서 생성).
    match /inviteCodes/{code} {
      allow get:   if signedIn();
      allow list:  if false;
      allow write: if false;
    }

    match /families/{fid} {
      allow read:   if isMember(fid);
      allow create: if signedIn() && request.resource.data.ownerUid == uid();
      allow update: if isOwner(fid);
      allow delete: if isOwner(fid);

      match /members/{memberUid} {
        allow read:   if isMember(fid);
        // 본인 합류(초대코드 검증 후) 또는 방장이 추가
        allow create: if signedIn() && (memberUid == uid() || isOwner(fid));
        allow update: if signedIn() && (memberUid == uid() || isOwner(fid));
        allow delete: if isOwner(fid) || memberUid == uid();
      }

      match /pets/{petId} {
        allow read, write: if isMember(fid);
      }

      match /pings/{pingId} {
        allow read:   if isMember(fid);
        allow create: if isMember(fid) && request.resource.data.memberUid == uid();
        // 댓글·리액션은 가족 누구나 가능하게 update 허용
        allow update: if isMember(fid);
        // 삭제는 본인 사진만
        allow delete: if isMember(fid) && resource.data.memberUid == uid();
      }

      match /rewards/{rewardId} { allow read, write: if isMember(fid); }
      match /pokes/{pokeId} {
        allow read:   if isMember(fid);
        allow create: if isMember(fid);
      }
      match /notifications/{notifId} { allow read, write: if isMember(fid); }
    }

    // FCM 토큰: 서버(Functions)만 읽고, 본인만 쓴다
    match /fcmTokens/{tokenUid} {
      allow read:  if false;
      allow write: if signedIn() && tokenUid == uid();
    }
  }
}
```

## 5. 스토리지 구조 & 규칙 (storage.rules)

경로: `families/{familyId}/pings/{date}/{uid}.jpg`, `families/{familyId}/mascot/{name}.png`

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    function isMember(fid) {
      return request.auth != null &&
        firestore.exists(/databases/(default)/documents/families/$(fid)/members/$(request.auth.uid));
    }
    match /families/{fid}/{allPaths=**} {
      allow read:  if isMember(fid);
      allow write: if isMember(fid)
                   && request.resource.size < 8 * 1024 * 1024
                   && request.resource.contentType.matches('image/.*');
    }
  }
}
```

## 6. 가입 · 합류 흐름

1. **가족 생성(방장)**: 익명 로그인 → 회원가입(이메일/구글) → `families/{fid}` 생성(ownerUid=내 uid) → `members/{uid}` 추가 → `users/{uid}.familyIds`에 fid 추가.
2. **초대코드 발급**: Cloud Function이 랜덤 코드 생성 → `inviteCodes/{code} = { familyId }`. (클라이언트가 임의 코드를 못 만들게 서버 전용)
3. **합류(구성원)**: 코드 입력 → `inviteCodes/{code}` 조회로 fid 획득 → 로그인 → `families/{fid}/members/{uid}` 생성 → `users/{uid}.familyIds`에 추가.

> 보안 팁: 초대코드는 만료시간(expiresAt)·1회성 옵션을 두면 안전. 합류 검증을 더 엄격히 하려면 Cloud Function `joinFamily(code)`에서 멤버 생성까지 처리.

## 7. 기존 코드(클라이언트) 변경 포인트

- `const FAMILY_ID = 'kimfamily'` → **로그인 후 결정되는 현재 가족 컨텍스트**(예: `currentFamilyId`)로 치환.
- 모든 Firestore 경로를 하위 컬렉션으로 변경:
  - `doc(db,'pings',id)` → `doc(db,'families',fid,'pings',id)`
  - `collection(db,'fcmTokens')` → 유지(최상위) + `familyId` 필드.
- `MEMBERS` 하드코딩 → `families/{fid}/members` 구독으로 동적 로딩.
- `DUBI_GOALS` / 점수 상수 → `families/{fid}.scoreRules` + `rewards` 컬렉션에서 로딩.
- 사진 업로드 경로 → `families/{fid}/pings/{date}/{uid}.jpg`.
- 점수 집계(`computeSeasonScores`)는 `families/{fid}/pings`를 시즌 범위로 쿼리하도록 수정.

## 8. 마이그레이션 계획 (단일 → 멀티)

1. `families/kimfamily` 문서 생성(ownerUid = 김아빠 uid, scoreRules 현재값 복사).
2. 기존 4명 → 첫 로그인 때 이름↔uid 매핑하여 `members/{uid}` 생성(임시 매핑 테이블 사용).
3. 기존 최상위 `pings/*` 문서를 `families/kimfamily/pings/*`로 복사(필드명 `withMembers`→사용 안 함, `withDubi`→`withPet` 등 정리).
4. Storage 기존 경로 → `families/kimfamily/...`로 이동(또는 URL 그대로 두고 점진 이전).
5. 검증 후 최상위 레거시 컬렉션 정리.

> 한 번에 끊지 말고, **새 구조로 쓰면서 과거 데이터는 읽기 호환**을 유지하는 방식이 안전.

---

## 9. 인덱스 · 비용 메모

- `pings` 시즌 집계: `date` 범위 쿼리 → `date` 단일 필드 인덱스(자동) 활용. 가족 하위 컬렉션이라 분산되어 비용 효율적.
- 읽기 절감: 점수 합계는 가족 문서에 캐시(`families/{fid}.scoreCache`)하고 핑 작성 시 Function이 증분 업데이트하는 방식도 가능(트래픽 커지면 도입).
- AI 마스코트 이미지: 생성 1회 → Storage 저장 후 URL 재사용(재호출 비용 방지).
